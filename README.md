# Проектная работа "Веб-ларек"

Стек: HTML, SCSS, TS, Vite

Структура проекта:

- src/ — исходные файлы проекта
- src/components/ — папка с JS компонентами
- src/components/base/ — папка с базовым кодом

Важные файлы:

- index.html — HTML-файл главной страницы

- src/utils/utils.ts — файл с утилитами

## Архитектура приложения

Код приложения разделен на слои согласно парадигме MVP (Model-View-Presenter), которая обеспечивает четкое разделение ответственности между классами слоев Model и View. Каждый слой несет свой смысл и ответственность:

Model - слой данных, отвечает за хранение и изменение данных.  
View - слой представления, отвечает за отображение данных на странице.  
Presenter - презентер содержит основную логику приложения и отвечает за связь представления и данных.

Взаимодействие между классами обеспечивается использованием событийно-ориентированного подхода. Модели и Представления генерируют события при изменении данных или взаимодействии пользователя с приложением, а Презентер обрабатывает эти события используя методы как Моделей, так и Представлений.

### Базовый код

#### Класс Component

Является базовым классом для всех компонентов интерфейса.
Класс является дженериком и принимает в переменной `T` тип данных, которые могут быть переданы в метод `render` для отображения.

Конструктор:  
`constructor(container: HTMLElement)` - принимает ссылку на DOM элемент за отображение, которого он отвечает.

Поля класса:  
`container: HTMLElement` - поле для хранения корневого DOM элемента компонента.

Методы класса:  
`render(data?: Partial<T>): HTMLElement` - Главный метод класса. Он принимает данные, которые необходимо отобразить в интерфейсе, записывает эти данные в поля класса и возвращает ссылку на DOM-элемент. Предполагается, что в классах, которые будут наследоваться от `Component` будут реализованы сеттеры для полей с данными, которые будут вызываться в момент вызова `render` и записывать данные в необходимые DOM элементы.  
`setImage(element: HTMLImageElement, src: string, alt?: string): void` - утилитарный метод для модификации DOM-элементов `<img>`

#### Класс Api

Содержит в себе базовую логику отправки запросов.

Конструктор:  
`constructor(baseUrl: string, options: RequestInit = {})` - В конструктор передается базовый адрес сервера и опциональный объект с заголовками запросов.

Поля класса:  
`baseUrl: string` - базовый адрес сервера  
`options: RequestInit` - объект с заголовками, которые будут использованы для запросов.

Методы:  
`get(uri: string): Promise<object>` - выполняет GET запрос на переданный в параметрах ендпоинт и возвращает промис с объектом, которым ответил сервер  
`post(uri: string, data: object, method: ApiPostMethods = 'POST'): Promise<object>` - принимает объект с данными, которые будут переданы в JSON в теле запроса, и отправляет эти данные на ендпоинт переданный как параметр при вызове метода. По умолчанию выполняется `POST` запрос, но метод запроса может быть переопределен заданием третьего параметра при вызове.  
`handleResponse(response: Response): Promise<object>` - защищенный метод проверяющий ответ сервера на корректность и возвращающий объект с данными полученный от сервера или отклоненный промис, в случае некорректных данных.

#### Класс EventEmitter

Брокер событий реализует паттерн "Наблюдатель", позволяющий отправлять события и подписываться на события, происходящие в системе. Класс используется для связи слоя данных и представления.

Конструктор класса не принимает параметров.

Поля класса:  
`_events: Map<string | RegExp, Set<Function>>)` - хранит коллекцию подписок на события. Ключи коллекции - названия событий или регулярное выражение, значения - коллекция функций обработчиков, которые будут вызваны при срабатывании события.

Методы класса:  
`on<T extends object>(event: EventName, callback: (data: T) => void): void` - подписка на событие, принимает название события и функцию обработчик.  
`emit<T extends object>(event: string, data?: T): void` - инициализация события. При вызове события в метод передается название события и объект с данными, который будет использован как аргумент для вызова обработчика.  
`trigger<T extends object>(event: string, context?: Partial<T>): (data: T) => void` - возвращает функцию, при вызове которой инициализируется требуемое в параметрах событие с передачей в него данных из второго параметра.

## Данные

В приложении используются следующие типы и интерфейсы данных:

### Типы

#### TPayment

```typescript
type TPayment = "card" | "cash";
```

Описывает способ оплаты заказа. Может принимать два значения:

- `card` - онлайн оплата картой
- `cash` - оплата при получении

#### TCategory

```typescript
type TCategory =
  | "софт-скил"
  | "хард-скил"
  | "кнопка"
  | "дополнительное"
  | "другое";
```

Определяет категорию товара для отображения с соответствующим стилем.

#### TValidationErrors

```typescript
type TValidationErrors = Partial<Record<keyof IBuyer, string>>;
```

Представляет объект с ошибками валидации полей покупателя. Ключи соответствуют полям интерфейса `IBuyer`, значения - текст ошибки. Если поле валидно, соответствующий ключ отсутствует.

### Интерфейсы

#### IProduct

```typescript
interface IProduct {
  id: string;
  description: string;
  image: string;
  title: string;
  category: TCategory;
  price: number | null;
}
```

Описывает товар в магазине. Содержит:

- `id` - уникальный идентификатор товара
- `description` - подробное описание товара
- `image` - имя файла изображения товара
- `title` - название товара
- `category` - категория товара
- `price` - цена товара в синапсах, может быть `null` для недоступных товаров

#### IBuyer

```typescript
interface IBuyer {
  payment: TPayment;
  email: string;
  phone: string;
  address: string;
}
```

Описывает данные покупателя, необходимые для оформления заказа:

- `payment` - способ оплаты
- `email` - электронная почта
- `phone` - номер телефона
- `address` - адрес доставки

#### IProductList

```typescript
interface IProductList {
  total: number;
  items: IProduct[];
}
```

Описывает ответ API при получении списка товаров:

- `total` - общее количество товаров
- `items` - массив товаров

#### IOrder

```typescript
interface IOrder extends IBuyer {
  total: number;
  items: string[];
}
```

Описывает заказ для отправки на сервер. Расширяет `IBuyer` и добавляет:

- `total` - общая стоимость заказа
- `items` - массив ID товаров в заказе

#### IOrderResult

```typescript
interface IOrderResult {
  id: string;
  total: number;
}
```

Описывает результат успешного оформления заказа:

- `id` - уникальный идентификатор созданного заказа
- `total` - итоговая сумма заказа

## Модели данных

Модели данных отвечают за хранение и логику работы с данными приложения.

### Класс ProductCatalog

Класс для управления каталогом товаров. Отвечает за хранение списка всех доступных товаров и товара, выбранного для предварительного просмотра.

Конструктор:  
`constructor(events: IEvents)` - принимает экземпляр брокера событий для генерации событий при изменении данных.

Поля класса:

- `protected _products: IProduct[]` - массив всех товаров в каталоге
- `protected _preview: string | null` - ID товара, выбранного для предварительного просмотра, или `null`
- `protected events: IEvents` - брокер событий

Геттеры:

- `get products(): IProduct[]` - возвращает массив всех товаров из каталога.
- `get preview(): string | null` - возвращает ID товара для предварительного просмотра или `null`.

Методы класса:

- `setProducts(products: IProduct[]): void` - сохраняет массив товаров, полученный в параметре, в поле `_products`. Генерирует событие об изменении каталога.
- `getProduct(id: string): IProduct | undefined` - возвращает товар по его ID или `undefined`, если товар не найден.
- `setPreview(product: IProduct): void` - сохраняет ID товара для предварительного просмотра. Генерирует событие об изменении превью.
- `getPreview(): IProduct | null` - возвращает товар, выбранный для предварительного просмотра, или `null`.

### Класс Basket

Класс для управления корзиной покупок. Отвечает за хранение товаров, выбранных пользователем для покупки, и операции с ними.

Конструктор:  
`constructor(events: IEvents)` - принимает экземпляр брокера событий для генерации событий при изменении корзины.

Поля класса:

- `protected _items: IProduct[]` - массив товаров в корзине
- `protected events: IEvents` - брокер событий

Геттеры:

- `get items(): IProduct[]` - возвращает массив товаров, находящихся в корзине.

Методы класса:

- `add(product: IProduct): void` - добавляет товар в корзину, если его там еще нет. Генерирует событие изменения корзины.
- `remove(productId: string): void` - удаляет товар из корзины по его ID. Генерирует событие изменения корзины.
- `clear(): void` - очищает корзину, удаляя все товары. Генерирует событие изменения корзины.
- `getTotal(): number` - вычисляет и возвращает общую стоимость всех товаров в корзине. Товары с `price = null` не учитываются.
- `getCount(): number` - возвращает количество товаров в корзине.
- `contains(productId: string): boolean` - проверяет, находится ли товар с указанным ID в корзине. Возвращает `true`, если товар есть в корзине, иначе `false`.

### Класс BuyerModel

Класс для управления данными покупателя. Отвечает за хранение информации о покупателе, необходимой для оформления заказа, и валидацию этих данных.

Конструктор:  
`constructor(events: IEvents)` - принимает экземпляр брокера событий для генерации событий при изменении или валидации данных.

Поля класса:

- `protected _payment: TPayment | null` - способ оплаты, выбранный покупателем, или `null`
- `protected _email: string` - электронная почта покупателя
- `protected _phone: string` - номер телефона покупателя
- `protected _address: string` - адрес доставки
- `protected events: IEvents` - брокер событий

Методы класса:

- `setField(field: keyof IBuyer, value: string): void` - устанавливает значение для указанного поля. Позволяет обновлять отдельные поля без изменения других. Генерирует событие изменения данных покупателя.
- `getData(): IBuyer` - возвращает объект со всеми данными покупателя в формате интерфейса `IBuyer`.
- `clear(): void` - очищает все данные покупателя, устанавливая начальные значения. Генерирует событие об очистке данных.
- `validate(): TValidationErrors` - выполняет валидацию всех полей покупателя согласно правилам:

  - `payment` - должен быть выбран (не `null`)
  - `address` - не должен быть пустой строкой
  - `email` - не должен быть пустой строкой
  - `phone` - не должен быть пустой строкой

  Возвращает объект `TValidationErrors`, где ключи - названия невалидных полей, значения - текст ошибки. Если все поля валидны, возвращает пустой объект.

## Слой коммуникации

Слой коммуникации отвечает за взаимодействие приложения с внешними сервисами, в частности с API сервера. Классы этого слоя инкапсулируют логику HTTP-запросов и преобразования данных.

### Класс WebLarekAPI

Класс для взаимодействия с API сервера Web-ларёк. Наследуется от базового класса `Api` и реализует специфичные для приложения методы работы с товарами и заказами.

Назначение и зона ответственности:  
Класс отвечает за все сетевые взаимодействия с сервером Web-ларёк. Он выполняет HTTP-запросы для получения каталога товаров, информации о конкретных товарах и отправки заказов. Класс также занимается формированием полных URL для изображений товаров, добавляя CDN префикс к относительным путям, полученным от сервера.

Конструктор:  
`constructor(cdn: string, baseUrl: string, options?: RequestInit)` - принимает:

- `cdn: string` - базовый URL CDN для получения изображений товаров
- `baseUrl: string` - базовый URL API сервера
- `options?: RequestInit` - опциональные настройки для запросов

Поля класса:

- `protected cdn: string` - хранит базовый URL CDN для формирования полных путей к изображениям товаров
- `readonly baseUrl: string` - (унаследовано от `Api`) базовый URL API сервера
- `protected options: RequestInit` - (унаследовано от `Api`) настройки для HTTP-запросов

Методы класса:

- `getProductList(): Promise<IProduct[]>` - выполняет GET запрос на эндпоинт `/product/` для получения списка всех товаров. Использует метод `get` базового класса `Api`. Возвращает промис с массивом товаров типа `IProduct[]`. Автоматически преобразует относительные пути изображений в полные URL, добавляя CDN префикс.
- `orderProducts(order: IOrder): Promise<IOrderResult>` - выполняет POST запрос на эндпоинт `/order/` для оформления заказа. Использует метод `post` базового класса `Api`. Принимает объект заказа типа `IOrder`, содержащий данные покупателя (способ оплаты, email, телефон, адрес), массив ID товаров и общую стоимость. Возвращает промис с результатом оформления заказа типа `IOrderResult` (ID созданного заказа и итоговую сумму).

Особенности реализации:  
Класс использует композицию, наследуясь от базового класса `Api` и переиспользуя его методы `get` и `post` для выполнения HTTP-запросов. Это обеспечивает единообразную обработку ошибок и работу с промисами. Преобразование путей к изображениям происходит на уровне этого класса, что позволяет моделям данных работать с полными URL без необходимости знать о CDN.

## Компоненты представления (View)

Компоненты представления отвечают за отображение данных в интерфейсе пользователя. Все компоненты наследуются от базового класса `Component` и используют сеттеры для обновления DOM-элементов. Компоненты не содержат бизнес-логики, а только управляют отображением и передают события в Презентер через callback-функции.

### Принципы проектирования View-компонентов:

1. **Один компонент — один блок разметки**: каждый класс представления отвечает за свой блок разметки и работает только с элементами внутри своего контейнера.

2. **Наследование для общего функционала**:

   - Три класса карточек (`CardCatalog`, `CardPreview`, `CardBasket`) наследуются от общего родителя `CardBase`, который содержит общую логику работы с названием и ценой товара.
   - Два класса форм (`OrderForm`, `ContactsForm`) наследуются от общего родителя `FormBase`, который содержит общую логику валидации и отправки форм.

3. **Модульность**: модальное окно `Modal` — это контейнер, который не имеет дочерних классов. Вся разметка, отображаемая в модальных окнах (карточки, корзина, формы), — это самостоятельные компоненты, которые могут быть выведены в любой блок сайта.

4. **Событийная модель**: в ответ на любое действие пользователя компоненты вызывают переданные им callback-функции, которые обрабатываются Презентером.

### Структура компонентов View:

| Компонент      | Наследуется от | Шаблон             | Назначение                              |
| -------------- | -------------- | ------------------ | --------------------------------------- |
| `Modal`        | `Component`    | `#modal-container` | Контейнер для модальных окон            |
| `Header`       | `Component`    | `.header`          | Шапка с корзиной и счётчиком            |
| `Gallery`      | `Component`    | `.gallery`         | Галерея товаров на главной странице     |
| `CardBase`     | `Component`    | —                  | Абстрактный родитель для карточек       |
| `CardCatalog`  | `CardBase`     | `#card-catalog`    | Карточка товара в каталоге              |
| `CardPreview`  | `CardBase`     | `#card-preview`    | Полная карточка товара в модальном окне |
| `CardBasket`   | `CardBase`     | `#card-basket`     | Карточка товара в корзине               |
| `FormBase`     | `Component`    | —                  | Абстрактный родитель для форм           |
| `OrderForm`    | `FormBase`     | `#order`           | Форма выбора оплаты и адреса (шаг 1)    |
| `ContactsForm` | `FormBase`     | `#contacts`        | Форма ввода email и телефона (шаг 2)    |
| `BasketView`   | `Component`    | `#basket`          | Компонент корзины                       |
| `Success`      | `Component`    | `#success`         | Окно успешного оформления заказа        |

### Класс Modal

Компонент модального окна. Управляет отображением модального окна, его открытием и закрытием.

Назначение:  
Отвечает за отображение модальных окон с различным контентом (карточки товаров, корзина, формы заказа, окно успеха).

Конструктор:  
`constructor(container: HTMLElement, onClose?: () => void)` - принимает:

- `container: HTMLElement` - корневой элемент модального окна
- `onClose?: () => void` - опциональный callback, вызываемый при закрытии модального окна

Поля класса:

- `protected _closeButton: HTMLButtonElement` - кнопка закрытия модального окна
- `protected _content: HTMLElement` - контейнер для содержимого модального окна
- `protected _onClose?: () => void` - callback для обработки закрытия

Сеттеры:

- `set content(value: HTMLElement)` - устанавливает контент модального окна

Методы класса:

- `open(): void` - открывает модальное окно, добавляет класс `modal_active` и навешивает обработчик нажатия Escape
- `close(): void` - закрывает модальное окно, удаляет класс `modal_active`, очищает контент и вызывает callback `onClose`
- `render(data: IModalData): HTMLElement` - рендерит модальное окно с переданным контентом и сразу открывает его

Особенности:

- Модальное окно закрывается по клику на overlay (вне контента)
- Закрывается по нажатию клавиши Escape
- Закрывается по клику на кнопку закрытия
- При закрытии контент очищается

### Класс Header

Компонент шапки страницы. Управляет отображением счётчика товаров в корзине и обработкой клика на кнопку корзины.

Назначение:  
Отображает логотип магазина и кнопку корзины с счётчиком количества товаров.

Конструктор:  
`constructor(container: HTMLElement, onBasketClick?: () => void)` - принимает:

- `container: HTMLElement` - корневой элемент шапки

## Слой Презентера

Презентер (реализован в `src/main.ts`) - это центральная логика приложения, которая связывает модели данных и компоненты представления. Презентер подписывается на события от моделей и обрабатывает их, а также получает уведомления от View через callback-функции.

### Архитектурный подход:

Презентер реализован **в основном скрипте приложения** (`main.ts`) без выделения в отдельный класс. Это упрощённый подход, подходящий для одностраничного приложения с единой логикой.

Код Презентера состоит из:

- **Обработчиков событий от моделей** (через `EventEmitter.on()`)
- **Функций для координации действий** (прямые вызовы без генерации событий)
- **Инициализации компонентов** и загрузки данных

### Важные принципы реализации:

1. **Презентер НЕ генерирует события** - он только обрабатывает события от моделей и выполняет действия напрямую
2. **Перерисовка View происходит только:**
   - При обработке событий от моделей данных (изменение данных)
   - При открытии модальных окон (действия пользователя)
3. **Вся бизнес-логика находится в Презентере:**
   - Валидация форм
   - Координация между моделями и представлениями
   - Управление потоком оформления заказа

### Основные обязанности Презентера:

1. **Инициализация приложения:**

   - Создание экземпляра брокера событий (`EventEmitter`)
   - Создание экземпляров моделей (`ProductCatalog`, `Basket`, `BuyerModel`, `WebLarekAPI`)
   - Создание экземпляров View-компонентов (`Modal`, `Header`, `Gallery`)
   - Получение ссылок на HTML шаблоны
   - Загрузка каталога товаров с сервера

2. **Обработка событий моделей данных:**

   - `catalog:changed` - создание карточек и отображение товаров в галерее
   - `preview:changed` - открытие модального окна с детальной информацией о товаре
   - `basket:changed` - обновление счётчика товаров в шапке

3. **Обработка действий пользователя (через callbacks):**

   - Клик на карточку товара → установка превью в модели
   - Клик на кнопку корзины → вызов `openBasket()`
   - Клик на "В корзину" / "Удалить" → вызов методов модели корзины
   - Клик на "Оформить" → вызов `openOrderForm()`
   - Отправка формы заказа → сохранение данных и вызов `openContactsForm()`
   - Отправка формы контактов → сохранение данных и вызов `submitOrder()`
   - Успешный ответ API → очистка моделей и вызов `showOrderSuccess()`

4. **Валидация данных форм:**

   - Валидация формы заказа (проверка способа оплаты и адреса)
   - Валидация формы контактов (проверка email и телефона)
   - Управление состоянием кнопок отправки форм
   - Отображение ошибок валидации

5. **Координация потока оформления заказа:**
   - `openBasket()` - открытие корзины в модальном окне
   - `openOrderForm()` - открытие первой формы (способ оплаты и адрес)
   - `openContactsForm()` - открытие второй формы (email и телефон)
   - `submitOrder()` - отправка заказа на сервер
   - `showOrderSuccess()` - отображение окна успешного заказа

## События приложения

В приложении используется событийно-ориентированная архитектура. События генерируются в трёх слоях: Models, View (через callbacks) и Presenter. Ниже приведён полный список событий с описанием.

### События моделей данных

События генерируются моделями при изменении данных с помощью `EventEmitter.emit()`.

| Событие           | Где генерируется                                    | Данные события                           | Описание                                                 |
| ----------------- | --------------------------------------------------- | ---------------------------------------- | -------------------------------------------------------- |
| `catalog:changed` | `ProductCatalog.setProducts()`                      | `{ products: IProduct[] }`               | Каталог товаров загружен или изменён                     |
| `preview:changed` | `ProductCatalog.setPreview()`                       | `{ product: IProduct }`                  | Выбран товар для предварительного просмотра              |
| `basket:changed`  | `Basket.add()`, `Basket.remove()`, `Basket.clear()` | `{ items: IProduct[] }`                  | Корзина изменилась (добавление/удаление товара, очистка) |
| `buyer:changed`   | `BuyerModel.setField()`                             | `{ field: keyof IBuyer, value: string }` | Изменено одно из полей данных покупателя                 |
| `buyer:cleared`   | `BuyerModel.clear()`                                | —                                        | Данные покупателя очищены                                |

### События View-компонентов

View-компоненты не генерируют события напрямую. Вместо этого они вызывают callback-функции, переданные в конструктор. Презентер, получая эти вызовы, генерирует события через `EventEmitter`.

| Компонент      | Callback              | Параметры                    | Когда вызывается                                               |
| -------------- | --------------------- | ---------------------------- | -------------------------------------------------------------- |
| `Header`       | `onBasketClick()`     | —                            | Клик на кнопку корзины                                         |
| `CardCatalog`  | `onClick(event)`      | `MouseEvent`                 | Клик на карточку товара в каталоге                             |
| `CardPreview`  | `onButtonClick()`     | —                            | Клик на кнопку "В корзину" / "Удалить из корзины"              |
| `CardBasket`   | `onDelete()`          | —                            | Клик на кнопку удаления товара из корзины                      |
| `BasketView`   | `onCheckout()`        | —                            | Клик на кнопку "Оформить"                                      |
| `OrderForm`    | `onSubmit(data)`      | `IOrderFormData`             | Отправка формы (клик "Далее")                                  |
| `OrderForm`    | `onInputChange(data)` | `Partial<IOrderFormData>`    | Изменение полей формы (способ оплаты или адрес)                |
| `ContactsForm` | `onSubmit(data)`      | `IContactsFormData`          | Отправка формы (клик "Оплатить")                               |
| `ContactsForm` | `onInputChange(data)` | `Partial<IContactsFormData>` | Изменение полей формы (email или телефон)                      |
| `Success`      | `onClose()`           | —                            | Клик на кнопку "За новыми покупками!"                          |
| `Modal`        | `onClose()`           | —                            | Закрытие модального окна (клик на крестик, overlay или Escape) |

### Координация в Презентере (без генерации событий)

Презентер **не генерирует события**. Вместо этого он использует прямые вызовы функций для координации действий:

| Функция                   | Когда вызывается                    | Что делает                                            |
| ------------------------- | ----------------------------------- | ----------------------------------------------------- |
| `openBasket()`            | Клик на кнопку корзины в Header     | Открывает корзину в модальном окне                    |
| `openOrderForm()`         | Клик на кнопку "Оформить" в корзине | Открывает первую форму заказа (способ оплаты и адрес) |
| `openContactsForm()`      | Отправка формы заказа               | Открывает вторую форму (email и телефон)              |
| `submitOrder()`           | Отправка формы контактов            | Отправляет заказ на сервер, очищает модели            |
| `showOrderSuccess(total)` | Успешный ответ от API               | Показывает окно успешного заказа                      |

**Принцип работы:** Презентер координирует взаимодействие компонентов через прямые вызовы функций, а не через события. Это упрощает логику и делает поток данных более явным.

### Поток данных в приложении:

```
Пользователь → View (callback) → Presenter (событие) → Model → Presenter (событие) → View (render)
```

Пример потока для добавления товара в корзину:

1. Пользователь кликает на карточку товара в каталоге
2. CardCatalog вызывает callback `onClick`
3. Презентер вызывает `catalogModel.setPreview(product)`
4. ProductCatalog генерирует событие `preview:changed`
5. Презентер обрабатывает событие и открывает модальное окно с CardPreview
6. Пользователь кликает на кнопку "В корзину" в CardPreview
7. CardPreview вызывает callback `onButtonClick`
8. Презентер вызывает `basketModel.add(product)`
9. Basket генерирует событие `basket:changed`
10. Презентер обрабатывает событие и обновляет счётчик в Header через `header.render({ counter: basketModel.getCount() })`
